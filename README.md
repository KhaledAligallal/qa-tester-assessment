# QA Tester Assessment — UTD Software

Welcome to the UTD Software QA Tester Technical Assessment.

This assessment evaluates your ability to test a **production-grade Laravel + Flutter stack** — the kind of challenges you'll face daily on our team: real-time features, payment webhooks, concurrency-sensitive endpoints, and a Flutter app shipping across multiple white-label flavors.

---

## Rules

- **AI usage is allowed.** Use ChatGPT, Copilot, Claude — whatever you want. However, several exercises are designed to require **human judgment, manual exploration, and personal experience** that AI cannot provide. We can tell the difference between a thoughtful answer and a generic AI response.
- **Time limit:** ~4 hours recommended across all exercises. No strict enforcement.
- **Partial submissions are welcome.** Quality over quantity — 5 excellent exercises beat 8 rushed ones.
- **Be honest.** Question 17 in the questionnaire asks how you used AI. There's no penalty — we value transparency.

## How to Complete This Assessment

### Step 1: Fork this repository
Click **"Fork"** on GitHub to create your own copy.

### Step 2: Clone and set up
```bash
git clone https://github.com/YOUR_USERNAME/qa-tester-assessment.git
cd qa-tester-assessment
```

**The app under test (Exercise 7) is hosted live — no clone or local server needed:**
👉 **https://mazen-salah.github.io/qa-tester-assessment/app/buggy-admin/**

You do **not** need a working Laravel or Flutter install. Exercises 3 and 5 are scoped so you can write your answers as-if — running them is optional. If you want to actually execute:
- **Exercise 3 (PHPUnit):** PHP 8.2+ and Composer
- **Exercise 5 (flutter_test):** Flutter 3.x+ and Dart 3.x+

### Step 3: Complete the exercises
All exercises are in the `exercises/` directory. There are **8 exercises** in 3 categories:

| # | Exercise | Type | Time | Points |
|---|----------|------|------|--------|
| 1 | Test Case Design — Concurrency-Sensitive Payment API | Written — Author test cases for a real-money endpoint | 30 min | 20 |
| 2 | Postman API Collection — Auth & Coins Flow | Code — Build chained Postman requests with assertions | 40 min | 20 |
| 3 | PHPUnit Feature Test — Sanctum-protected API | Code — Fix a broken Laravel Feature test | 30 min | 20 |
| 4 | Selenium Refactor — Login Automation | Code — Refactor anti-patterns into POM + waits | 20 min | 20 |
| 5 | Flutter Widget Test — Login Form | Code — Write flutter_test widget tests from scratch | 30 min | 20 |
| 6 | Video Walkthrough — Live Bug Triage | Video — Record yourself testing an app aloud | 20 min | 20 |
| 7 | Visual Bug Hunt — Buggy Admin Panel | Run & Report — Find bugs by clicking around | 25 min | 20 |
| 8 | QA Strategy Decisions | Written — Real-world trade-off scenarios | 30 min | 20 |

**Exercises 6-8 are AI-resistant** — they require running the app, recording video, and making judgment calls that AI cannot do well.

### Step 4: Complete the questionnaire
1. Open `questionnaire/index.html` in your browser
2. Fill out all sections (20 questions)
3. Click **"Generate Answers File"** — downloads `questionnaire_answers.json`
4. Move it to the `answers/` directory

### Step 5: Commit and push
```bash
git add .
git commit -m "Complete assessment"
git push origin main
```

### Step 6: Submit
Reply to the assessment email with your **forked repo URL**.

If your repo is private, add the GitHub username provided by your hiring contact as a collaborator.

## What We Evaluate

| Dimension | What We Look For |
|-----------|-----------------|
| **Test Design** | Coverage of happy path, edge cases, negative paths, concurrency, security |
| **Automation Quality** | POM, proper waits, parameterization, no `Thread.sleep`, maintainable selectors |
| **API Testing Depth** | Chained requests, env vars, schema validation, status + body + header assertions |
| **Bug Reporting** | Clear repro steps, severity reasoning, evidence (screenshots), proposed fix |
| **Stack Awareness** | Understanding Laravel/Sanctum auth, Flutter widget tree, MySQL transactions |
| **QA Judgment** | Trade-off analysis, realistic plans, what NOT to test (Exercise 8) |
| **Communication** | Clear explanations, well-structured written answers, would convince a PM/dev |

## FAQ

**Q: Do I need to complete all 8 exercises?**
No. Complete as many as you can with high quality. We'd rather see 5 excellent exercises than 8 rushed ones.

**Q: Can I use AI tools?**
Yes. We encourage using whatever tools you'd use in your daily work. However, exercises 6-8 specifically require human judgment — video recording, running the app, and QA trade-off decisions that AI handles poorly.

**Q: Do I need a working Laravel / Flutter / Selenium environment?**
Not strictly. Exercises 3, 4, and 5 can be answered as written code without execution — we evaluate the code, not the runtime output. But if you DO run them and include a screenshot of passing tests, that's a strong bonus signal.

**Q: What tools do I need installed for Exercise 7?**
Just a modern browser. Open `app/buggy-admin/index.html` directly — it's self-contained, no server required.

**Q: Can I use a different tool than the one specified?**
Exercises 2-5 ask for specific tools (Postman, PHPUnit, Selenium, flutter_test) because we use them on our team. If you have a strong reason to substitute (e.g., REST Assured instead of Postman), do so and note why in the file's header comment.

**Q: What if I find a bug in the assessment itself?**
Great eye! Note it in your README or questionnaire — we'll consider it a strong signal.

**Q: How long do I have?**
5 business days from when you receive the assessment email.
